## 5. Run protocol

For each variant:

1. **Cold-cache reset**: delete `/tmp/lme-bench-*.db`, restart `ollama`.
2. **Warmup**: 3 full passes (500 questions each) — discarded.
3. **Measurement**: 5 full passes — record per-pass R@1/R@5/R@10/R@20.
4. **Aggregate**: report **median** across the 5 measurement passes.
5. **Cool-down**: ≥5 minutes between variants (M2 thermal recovery).

Latency is measured wall-clock with `time.monotonic()` around the recall
call only (storage time is excluded — it's a one-shot setup cost).

## 7. ai-memory configuration per variant

All variants share these defaults (from `~/.config/ai-memory/config.toml`
generated by the harness — not the user's config):

```toml
[storage]
db_path = "/tmp/lme-bench-${VARIANT}.db"
archive_on_gc = true

[recall]
# Adaptive blend: semantic_weight 0.50 (≤500 chars) → 0.15 (≥5000 chars)
adaptive_blend = true
semantic_threshold = 0.20
```

Per-variant overrides are applied via `--features` and `--tier` CLI flags,
documented in `run_variants.sh`.

## 8. Metrics

- **R@K** (Recall at K): fraction of questions where the correct source
  session id appears in the top K recalled memories.
- **q/s**: queries per second, wall-clock, single-threaded recall path
  (parallel runs annotated separately).
- **Median**: 5 measurement passes, sorted, middle value (3rd of 5).
- **Median P95 latency**: per-question wall-clock recall latency, 95th
  percentile, taken from the median pass.

We report **R@5** as the primary number because it matches the LongMemEval
paper's headline metric. R@1 and R@10/R@20 are reported as a sanity check
on the curve shape.

---

## 9. Honesty notes

- **No retraining**: variants disclose the existing range of the v0.6.3
  recall pipeline. We do not tune scoring weights to chase a higher number
  for this disclosure.
- **No oracle leakage**: harnesses do not see the ground-truth session id
  during recall. They only see the question text and the corpus.
- **No averaging across passes hides regression**: the median is reported,
  but per-pass numbers are saved to `results/raw-${VARIANT}-${PASS}.csv`
  and committed for audit.
- **Floor matters**: keyword-baseline is the *floor* (no embedding cost,
  no LLM cost). Any variant that doesn't meaningfully exceed the floor
  isn't worth its compute budget — we say so explicitly in `results.md`.

---

## 10. Known sources of variance

| Source | Magnitude | Mitigation |
|---|---|---|
| Ollama JIT warm vs cold | ±0.4% R@5 | 3 warmup passes |
| FTS5 cache state | ±0.2% R@5 | Same DB across passes |
| M2 thermal throttling | ±15% q/s | 5 min cool-down between variants |
| Floating-point reduction order | <0.1% R@5 | Single-thread fallback for reproducibility audit |
| Ollama model digest drift | could shift R@5 ±2% | Pin digest; verify before each run |

Any cell in `results.md` marked **PENDING-RUN** has methodology fully
nailed down here but compute deferred to a later operator session on the
reference hardware. The runner script `run_variants.sh` will populate
those cells as runs complete.
