For developers

Build with ai-memory.

Three interfaces share one SQLite-backed core: MCP stdio, HTTP REST, native CLI. Same data, same validation, same agent_id semantics. This page gives you a working example of each in ten lines or less, then points at the extension surfaces.

Build from source

Rust 1.88+, one binary.

git clone https://github.com/alphaonedev/ai-memory-mcp
cd ai-memory-mcp
cargo build --release
# binary at target/release/ai-memory (~26 MB stripped)

All four gates must pass before a PR is opened:

cargo fmt --check
cargo clippy -- -D warnings -D clippy::all -D clippy::pedantic
AI_MEMORY_NO_CONFIG=1 cargo test
cargo audit
Three interfaces, one example each

Pick the surface that fits your call site.

All three share src/db.rs + src/validate.rs. There is no behavioural divergence; whatever the CLI does, the MCP tool and the HTTP endpoint do too.

printf '%s\n' \

  '{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"memory_store","arguments":{"title":"hello","content":"first MCP memory","tier":"mid"}}}' \

NHI agent_id semantics

Every memory carries a claimed identity.

metadata.agent_id is the AI Non-Human Identity marker. It is a claimed identity, not an attested one — pair it with agent registration before using it for security decisions.

Resolution ladder (CLI + MCP)

  1. Explicit caller value (--agent-id, MCP agent_id param, or metadata.agent_id in store request)
  2. AI_MEMORY_AGENT_ID env var
  3. (MCP only) initialize.clientInfo.name → ai:<client>@<hostname>:pid-<pid>
  4. host:<hostname>:pid-<pid>-<uuid8> (stable per-process)
  5. anonymous:pid-<pid>-<uuid8> (no hostname)

The HTTP daemon is multi-tenant and skips the process-level default: it reads agent_id from the request body, then X-Agent-Id, then assigns anonymous:req-<uuid8> and logs WARN.

Validation regex: ^[A-Za-z0-9_\-:@./]{1,128}$. Permits prefixed forms (ai:, host:), @ scope separator, / for future SPIFFE-style ids. Rejects whitespace, null bytes, control chars, shell metacharacters.

Full reference (immutability invariants, special metadata keys, defaults-that-leak warnings): agent-identity.html.

Hooks and subscriptions

20-event hook pipeline + HMAC subscriptions.

The hook pipeline fires on 20 named substrate events: memory_store, memory_touch, memory_link, memory_reflect, kg_invalidate, etc. Hooks run synchronously (default) or async-fan-out (subscriptions).

Register a webhook subscription

ai-memory subscribe \
  --event memory_store \
  --url https://my-app.example.com/ai-memory-events \
  --hmac-secret-file /etc/my-app/hmac.key

Events are POSTed as JSON with an X-AI-Memory-Signature HMAC header. Unsigned subscribers are refused at registration. SSRF gate: loopback URLs require AI_MEMORY_ALLOW_LOOPBACK_WEBHOOKS=1.

Full hook taxonomy + payload schemas: hook-pipeline.md.

Extend the substrate

New tools, new families, new hooks.

New MCP tool

  1. Add JSON definition in tool_definitions() (src/mcp.rs)
  2. Add match arm in the dispatch block
  3. Implement handler: fn(&Connection, params) -> Result<Value>

New CLI subcommand

  1. Add variant to Command enum (src/main.rs)
  2. Define an Args struct
  3. Add dispatch case in main()
  4. Implement cmd_* handler taking &Path + args

New HTTP endpoint

  1. Add route in the Axum router (src/main.rs)
  2. Implement handler in handlers.rs using the Db extractor

For deeper changes (new memory_kind, new family, new tier), the writeup is in DEVELOPER_GUIDE.md and the family/kind distinction is clarified in agent-skills.md.
