# ai-memory v0.6.3.1 — the never-lose-context release

> Persistent memory for any AI. Self-hosted. MCP-native. Tamper-evident.

---

## TL;DR by audience

### 👤 If you're a non-technical user

You ever notice how every time you open Claude / Cursor / your AI assistant, it forgets everything you talked about yesterday? That ends today.

ai-memory v0.6.3.1 is a tiny background service you install once. Every AI session you open from then on starts already aware of what you and the AI worked on previously. Your AI picks up exactly where the last conversation left off — automatically, no special prompting, no copy-paste.

It's local. Your memory lives in a single file on your computer. Nothing goes to the cloud. Works with Claude, ChatGPT, Grok, Gemini, and every other AI assistant we know about.

**One command to install:**
```
brew install ai-memory && ai-memory install claude-code --apply
```

That's it. Restart Claude Code. It now remembers.

### 🏢 If you're a C-level decision maker

ai-memory is enterprise-class memory infrastructure for AI agents. v0.6.3.1 ships:

- **Continuity** — agents resume work across sessions, devices, and even across vendors (Claude → GPT → Grok → Gemini all share the same store)
- **Auditability** — opt-in tamper-evident hash-chained audit log, SIEM-ready JSON ingest (Splunk / Datadog / Elastic / Loki recipes shipping in `docs/security/`)
- **Compliance** — preset configurations for SOC 2 / HIPAA / GDPR / FedRAMP retention and redaction policies
- **Self-hosted** — zero cloud dependencies, your data stays in your infrastructure (laptop / server / Kubernetes pod / on-prem)
- **Cross-platform** — macOS / Linux / Windows / Docker / Kubernetes / ARM Linux / commercial Unix / embedded Linux. Single binary, same store shape, every platform
- **Apache 2.0** — use it however you want; no vendor lock-in

The total cost of ownership is "the disk space your team uses" plus "the engineering hours to write the SessionStart hook in their AI host" (one config-file edit, automated by `ai-memory install`).

For regulated environments, the audit trail is the document of record. Hash-chained line-by-line, append-only at the OS level where supported, signed checkpoint attestation reserved for v0.7+.

This is the release that elevates ai-memory from "nifty open-source project" to enterprise memory infrastructure for any AI deployment.

## Migration notes

Upgrading from v0.6.3 / v0.6.3+patch.x: schema migrates automatically on first run (v15 → v17 → v18 → v19). All existing memories preserved. The migration is one-way; back up your DB before upgrading if you need to roll back.

```
cp ~/.claude/ai-memory.db ~/.claude/ai-memory.db.pre-v0631.bak
```

If you have the v0.6.3 SessionStart hook (or no hook), `ai-memory install claude-code --apply` is idempotent — runs cleanly.

## All new features in v0.6.3.1

### CLI surfaces (5 new subcommands)

1. **`ai-memory boot`** — universal session-boot context primitive. Read-only, fast (no embedder, no daemon, indexed list only). Always emits a 5-field diagnostic manifest (`version`, `db` with schema + memory count, `tier` with embedder/reranker/llm, `latency`, `namespace` with loaded count). Four status variants: `ok` / `info-fallback` / `info-empty` / `warn`. Three output formats: `text` (default) / `json` / `toon`. Flags: `--namespace`, `--limit` (clamped 1..50), `--budget-tokens`, `--format`, `--no-header`, `--quiet`, `--cwd`. Exits 0 in every state — never wedges an agent's first turn.

2. **`ai-memory install <agent>`** — multi-target config-file installer. 6 targets: `claude-code`, `openclaw`, `cursor`, `cline`, `continue`, `windsurf`. `--dry-run` default with unified-diff output, `--apply` opt-in, `--uninstall --apply` removes the managed block precisely. Idempotent marker block (`// ai-memory:managed-block:start/end` + `managed-keys` allowlist). JSON roundtrip validation before write. `.bak.<rfc3339>` backup file. World-writable destination refused. Some targets accept `--config <path>` override where canonical paths aren't auto-discoverable upstream.

3. **`ai-memory wrap <agent>`** — cross-platform Rust replacement for bash/PowerShell shell-wrapper recipes. Spawns the named CLI with `ai-memory boot` context delivered via the appropriate strategy: `SystemFlag` (default `--system <msg>`), `SystemEnv` (env var injection), `MessageFile` (tempfile + `--message-file <path>`), `Auto` (per-agent lookup table). Built-in defaults: `codex`/`codex-cli` → SystemFlag, `gemini` → SystemFlag, `aider` → MessageFile, `ollama` → SystemEnv, fall-through `--system`. Same binary works on macOS / Linux / Windows / Docker / Kubernetes. Exit code propagated.

4. **`ai-memory logs`** — operator CLI for the operational logging facility. Subcommands: `tail [--follow]`, `cat`, `archive` (zstd-compress files past retention), `purge --before <date>` (with audit-gap warning). Filters: `--since`, `--until`, `--level`, `--namespace`, `--actor`, `--action`, `--format text|json`. Default-OFF; opt-in via `[logging] enabled = true` in config.toml.

5. **`ai-memory audit verify | tail | path`** — operator CLI for the security audit trail. `verify` walks the hash chain, exits 0 on integrity / 2 on tamper detection. `tail [N]` prints recent events. `path` prints the resolved audit log path. Default-OFF; opt-in via `[audit] enabled = true`.

### Core capabilities

6. **Tamper-evident security audit log** — separate channel from operational logs. Every memory operation (recall / store / update / delete / link / promote / forget / consolidate / export / import / approve / reject / session_boot) emits a JSON event with: `schema_version`, `timestamp`, `sequence`, `actor` (agent_id + scope + NHI synthesis), `action`, `target` (memory_id + namespace + scope), `outcome` (allow/deny/error/pending), `auth` (source_ip + mtls_fp + api_key_id_hash), `session_id`, `request_id`, `error`, `prev_hash`, `self_hash`. Hash-chained line-by-line. OS append-only hint (`chattr +a` on Linux, `fs_chflags` UF_APPEND on macOS/BSD). Schema v1, stable + versioned + documented in `docs/security/audit-trail.md`.

7. **Operational logging facility** — `tracing-appender` rolling file appender for the existing `tracing::info!` / `tracing::warn!` call sites (llm.rs, subscriptions.rs, etc.). Rotation by size + time. Retention/archival via zstd compression. Optional structured JSON for SIEM ingestion (Splunk / Datadog / Elastic / Loki recipes shipping in `docs/security/`).

8. **Compliance presets** — config-block presets for SOC 2 / HIPAA / GDPR / FedRAMP. Each sets retention, redaction, attestation cadence, encrypt-at-rest hint, pseudonymize-actors flag (GDPR). Activate via `[audit.compliance.<frame>] applied = true`.

9. **User-configurable log + audit paths** at every layer with documented precedence: CLI flag (`--log-dir` / `--audit-dir`) > env var (`AI_MEMORY_LOG_DIR` / `AI_MEMORY_AUDIT_DIR`) > config.toml > platform default (Linux: `${XDG_STATE_HOME}/ai-memory/`, macOS: `~/Library/Logs/ai-memory/`, Windows: `%LOCALAPPDATA%\ai-memory\`, systemd: `/var/log/ai-memory/`). 0700 mode on Unix; refuses world-writable destinations.

10. **Boot version-drift detection** — `MIN_SUPPORTED_SCHEMA` / `MAX_SUPPORTED_SCHEMA` constants in boot manifest. New `BootStatus::WarnSchemaUnsupported` variant + `schema_supported: bool` JSON flag. Boot detects DB schema outside the binary's supported range and emits `# ai-memory boot: warn — db schema vN unsupported by binary X.Y.Z`. Boot still exits 0 — never wedges the agent.

11. **`[boot]` privacy opt-out** — new config block:
    - `[boot] enabled = false` → silent exit 0, empty stdout, no manifest. Privacy-sensitive contexts that don't want memory titles in CI logs.
    - `[boot] redact_titles = true` → manifest still appears, but body rows redact titles to `<redacted>`. Compliance contexts that need audit trail of "boot ran with N memories" without exposing memory subjects.
    - `AI_MEMORY_BOOT_ENABLED=0` env var override (highest precedence).

### Internals

18. **`ai-memory boot` plumbed through `&AppConfig`** so the manifest can resolve the configured tier (embedder / reranker / llm) without instantiating them. Boot stays fast — no model loading, no daemon startup.

19. **Always-on diagnostic header** (no blackbox) — `--quiet` only suppresses stderr; the stdout status header always appears so the agent and human always see what loaded and why. `--no-header` exists for advanced wrappers but is explicitly NOT recommended for production hooks (silent failure becomes indistinguishable from "no memories yet").

20. **Cross-platform `tempfile` promoted to main deps** (was dev-only) for the `MessageFile` strategy in `ai-memory wrap`.
