Features

🧠

Three Memory Tiers

Short-term (6h), mid-term (7d), and long-term (permanent) — mimicking how human memory works. Expired memories are automatically garbage collected.

🔍

Full-Text Search

SQLite FTS5 with BM25 ranking, priority boosting, and access frequency weighting. Find what matters fast.

📡

HTTP API + CLI

Axum-powered REST API on localhost:9077 plus a full CLI for direct terminal access. Both interfaces, same database.

🎯

/recall Endpoint

The primary "what do I know about X?" interface. Searches across all tiers, boosts long-term and frequently-accessed memories.

📁

Free-form Namespaces

Organize memories by project, repo, topic — any string works. Filter queries by namespace to scope results.

⚡

Zero-Dependency Runtime

Single static binary. SQLite bundled and compiled in. WAL mode for concurrent reads. One file is the entire database.

Architecture

claude-memory models its storage after human memory systems. Each memory has a tier that determines its lifetime, plus metadata for intelligent ranking.

short
TTL: 6 hours

Ephemeral session context. Current task details, scratch notes, temporary state. Auto-expires.

mid
TTL: 7 days

Working knowledge. Sprint context, recent decisions, active investigation notes. Default tier.

long
TTL: permanent

Durable knowledge. Architecture decisions, project conventions, hard-won lessons. Never expires.

Recall Scoring

The /recall endpoint combines multiple signals to surface the most relevant memories:

score = BM25(query) + (priority * 0.5) + (access_count * 0.1) + tier_boost

Where tier_boost is: long = 3.0, mid = 1.0, short = 0.0
Frequently accessed, high-priority, long-term memories float to the top.

Storage

SQLite in WAL mode with bundled compilation — no system SQLite dependency. The database is a single portable file. FTS5 virtual table with triggers keeps the full-text index in sync automatically. Access counts and recency are tracked on every read.

Quick Start

Build from source

Requires Rust 1.70+. Clone the repo and build in release mode:

git clone https://github.com/alphaonedev/claude-memory.git
cd claude-memory
cargo build --release

Start the daemon

Launch the HTTP server on 127.0.0.1:9077:

./target/release/claude-memory serve

Custom host/port and database path:

./target/release/claude-memory --db ~/my-memories.db serve --port 8080

Store a memory

Via the HTTP API:

curl -X POST http://127.0.0.1:9077/api/v1/memories \
  -H "Content-Type: application/json" \
  -d '{
    "tier": "long",
    "namespace": "my-project",
    "title": "Database schema uses UUID primary keys",
    "content": "All tables use UUIDv4 as primary keys. Never use auto-increment.",
    "tags": ["architecture", "database"],
    "priority": 8
  }'

Or via the CLI:

claude-memory store \
  -t long -n my-project -T "UUID primary keys" \
  -c "All tables use UUIDv4 as primary keys." \
  --tags architecture,database -p 8

Recall what you know

Ask the memory system what it knows about a topic:

# HTTP
curl "http://127.0.0.1:9077/api/v1/recall?context=database+primary+keys&namespace=my-project"

# CLI
claude-memory recall "database primary keys" -n my-project

API Reference

All endpoints are prefixed with /api/v1. The daemon listens on 127.0.0.1:9077 by default.

Method Endpoint Description
GET /api/v1/health Health check. Returns {"status": "ok"}
GET /api/v1/recall Recall memories by context. Params: context (required), namespace, limit
GET /api/v1/search Full-text search. Params: q (required), namespace, tier, limit, min_priority
GET /api/v1/memories List memories. Params: namespace, tier, limit, offset, min_priority
POST /api/v1/memories Create a memory. Body: title, content (required); tier, namespace, tags, priority, ttl_secs
POST /api/v1/memories/bulk Bulk create. Body: array of memory objects
GET /api/v1/memories/{id} Get a single memory by ID. Bumps access count.
PUT /api/v1/memories/{id} Update a memory. Body: any subset of fields
DELETE /api/v1/memories/{id} Delete a memory by ID
GET /api/v1/stats Database statistics: counts by tier and namespace, DB size
POST /api/v1/gc Trigger garbage collection of expired memories

CLI Reference

Every operation available via the API is also accessible from the command line. Global flag: --db <path> (default: claude-memory.db, env: CLAUDE_MEMORY_DB).

serve

Start the HTTP daemon.

claude-memory serve [--host 127.0.0.1] [--port 9077]

store

Store a new memory.

claude-memory store -T "Title" -c "Content body" \
  [-t short|mid|long] [-n namespace] [--tags a,b,c] [-p 1-10]

recall

Recall memories relevant to a context string. The primary retrieval interface.

claude-memory recall "what are the database conventions" \
  [-n namespace] [--limit 10]

search

Full-text search across all memories.

claude-memory search "UUID primary key" \
  [-n namespace] [-t tier] [--limit 20]

get

Retrieve a single memory by ID.

claude-memory get <uuid>

list

List stored memories with optional filters.

claude-memory list [-n namespace] [-t tier] [--limit 20]

delete

Delete a memory by ID.

claude-memory delete <uuid>

gc

Run garbage collection — removes all expired short-term and mid-term memories.

claude-memory gc

stats

Show database statistics: total count, breakdown by tier and namespace, database file size.

claude-memory stats
